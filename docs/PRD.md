# 保活手帳 - プロダクト仕様定義書

## 概要

本ドキュメントは、保活手帳のプロダクト版制作に向けた人間中心設計（HCD）に基づく仕様定義である。現プロトタイプの検証と開発者自身の保活体験の深掘りを通じて、解決すべき本質的な課題とプロダクトの方向性を整理した。

---

## 1. 開発方針：SLC（Simple, Lovable, Complete）

### MVPではなくSLCを採用する理由

参考：[Your customers hate MVPs. Make a SLC instead.](https://longform.asmartbear.com/slc/)

MVPは「最小限の機能で早く出して学ぶ」アプローチだが、ユーザーに未完成品を使わせることになる。SLCは機能が少なくても「完成している」と感じられる体験を提供する。

| 観点 | MVP | SLC |
|------|-----|-----|
| Simple | 機能を削る | 機能を削る |
| Viable / Lovable | 「使える」レベル | 「使いたい」レベル |
| Minimum / Complete | 未完成でもOK | その範囲では完成している |

### SLCの3要素（保活手帳の場合）

**Simple（シンプル）**
- 機能は最小限
- 複雑な設定や操作がない
- 迷わず使える

**Lovable（愛される）**
- 見学中に片手でサッと使える
- 「これいいな、また使いたい」と感じる
- スプシや紙より明らかに良い体験

**Complete（完結している）**
- 「園の印象を残して、後から思い出せる」という目的が達成できる
- 機能が少なくても「これで十分」と感じる
- 追加機能がなくても価値がある

## 2. SLC v1 の定義

### v1でユーザーが得られる体験

**「あの園はこういう特徴だった」を、メモを読んで「そうそう、こういうとこだった！」って思える**

### v1の機能

| 機能 | 説明 |
|------|------|
| 園を追加できる | 園名を入力して追加 |
| 見学日を記録できる | いつ見学したか／する予定か（任意、「未定」も可） |
| 園ごとに自由メモを書ける | 印象、気づき、特徴を自由に記入 |
| 後から見返せる | 園の一覧からメモを確認できる |
| 編集できる | 園名、見学日、メモをそれぞれ編集できる |
| 削除できる | 誤登録した園を削除できる（確認ダイアログあり） |
| 初回ヒント | 見学のコツ（3つ）を初回起動時に表示 |

### v1に含めないもの

| 機能 | 理由 | 将来バージョン |
|------|------|---------------|
| 質問テンプレート | なくても「印象を残す」は達成できる | v2候補 |
| はい・いいえ選択式 | なくても「印象を残す」は達成できる | v2候補 |
| 振り返りガイド | なくても「印象を残す」は達成できる | v2候補 |
| 並び替え機能 | v1は追加順固定で十分 | v2候補 |
| 比較機能 | 優先度低い（直感で判断される） | v3以降 |
| パートナー共有 | 有料機能として検討 | 将来 |
| データエクスポート | v1は最小限に | v1.1候補 |

### v1の「Complete」の判定基準

以下がすべて「はい」なら、v1はCompleteである。

- [ ] 見学中に実際に使えたか？
- [ ] 片手で操作できたか？
- [ ] 後から見返して記憶が蘇ったか？

### v1の「Lovable」の判定基準

- [ ] スプシより使いやすいと感じたか？
- [ ] 次の見学でも使いたいと思ったか？
- [ ] 人に勧めたいと思ったか？

## 3. 本質的な課題

### 発見された課題

**「複数園を見学した後、申込書を書く段階で『あの園どんな雰囲気だったっけ？』と記憶が曖昧になり、判断に困る」**

### 課題の背景

- 市役所から「できるだけたくさん見学して希望園を埋めろ」と言われる
- 見学は5〜10園程度、期間は数ヶ月にわたる
- 見学時にメモを取ろうとしても、スプレッドシートはスマホで使いにくく破綻する
- 紙にメモしても字が汚くて読み返す気にならない
- 結果、申込時に記憶が曖昧なまま希望順位を決めることになる

### 重要な洞察

**最終判断は「直感」で行われる。しかしその直感の質は「見学で何を見て、何を感じたか」の記憶に依存する。**

つまり、記録の目的は「比較検討」よりも **「思い出すため」** である。

## 5. ユーザージャーニー

| フェーズ | ユーザーの状態 | 課題 | v1の役割 |
|---------|---------------|------|----------|
| **見学前** | 通える範囲の園をリストアップ、見学予約 | 何を見ればいいかわからない | 園を追加してリスト化、初回ヒント（見学のコツ3つ） |
| **見学中** | 説明を聞きながら、気になることをメモしたい | スプシは使えない、紙は読み返せない | サッと開いて、印象・気づきを残せる |
| **見学直後** | 「どうだった？」をパートナーと話したい | 記憶が鮮明なうちに整理したい | メモを見ながら会話できる |
| **申込時** | 複数園の記憶が曖昧、希望順位を決めたい | 「あの園どうだったっけ？」となる | 園ごとのメモを見返し、思い出す |

### ユーザーの典型的な行動フロー（v1想定）

1. **リストアップ期**：通える範囲の園を保活手帳に追加（園名のみ）
2. **予約期**：電話/メールで見学日を調整 → 見学日を追記
3. **見学当日**：見学中または直後にメモを書く
4. **見学後**：必要に応じてメモを追記・編集
5. **申込時**：一覧からメモを見返して記憶を呼び起こす

## 6. プロダクトのコア価値

### バリュープロポジション

**「見学の記憶を、申込時まで鮮度高く残す」**

- 比較表を作ることでも、質問を完璧にこなすことでもない
- 「あのとき感じたこと」を後から思い出せるようにすること

### 現プロトタイプとのギャップ

| 観点 | 現プロトタイプ | 新しい方向性 |
|------|---------------|-------------|
| 核となる機能 | 質問と回答の記録 | 印象・気づきのメモ |
| 質問テンプレート | 主役 | 補助（聞き忘れ防止） |
| 記録の目的 | 比較検討のため | 思い出すため |
| 比較機能 | 想定していた（未実装） | 優先度低い |

## 7. 質問の2分類

ユーザーインタビューから、見学時の質問は2種類に分けられることが判明した。

### 種類A：運用・実務的な質問（スペック確認）

入園後の生活がどうなるかを知るための質問。正解・不正解がある。

**例：**
- 連絡帳は手書き？アプリ？
- おむつは持参？サブスク？
- 駐車場・駐輪場はある？
- お昼寝布団は必要？コット？
- 土曜保育のルールは？
- 投薬はしてもらえる？

### 種類B：雰囲気・方針の確認（フィーリング確認）

「この園に預けたいか」を判断するための観察。正解がなく、直感に影響する。

**例：**
- 先生同士の会話のトーン
- 子どもが泣いているときの先生の対応
- 電話予約時の対応（ガチャ切りされないか）
- 保育士がカリカリしていないか
- 古い保育感覚ではなさそうか（研修をしているか）

### 重要な洞察

**質問リストはスペック確認には有効だが、最終判断に本当に影響するのは「言語化しにくい観察・直感」の方である。**

## 8. 機能ロードマップ

### v1（SLC）：印象を残す

#### 画面一覧

| 画面 | 役割 |
|------|------|
| 初回オンボーディング | 見学のコツ（3つ）を表示、初回のみ |
| 園一覧（ホーム） | 登録した園を見返す、園追加画面への導線 |
| 園追加 | 園名と見学日を入力して追加 |
| 園詳細 | メモを見る・編集する、園情報を編集・削除する |

#### 「片手でサッと」の定量基準

| 操作 | 目標タップ数 | 目標時間 |
|------|-------------|---------|
| 起動 → 園追加画面 | 2タップ以内 | - |
| 園追加（園名入力→保存） | - | 15秒以内 |
| 起動 → 既存園のメモ追記開始 | 2タップ以内 | - |
| メモ保存 | 自動保存 or 1タップ | - |

- 片手持ちで親指が届く範囲に集約

#### 編集・削除・並び順の仕様

| 操作 | 仕様 |
|------|------|
| 園名の編集 | インライン編集、空にはできない |
| 見学日の編集 | 日付ピッカーまたは「未定」を選択 |
| メモの編集 | インライン編集、複数行対応、空でもOK |
| 園の削除 | 確認ダイアログ「この園を削除しますか？」→ 削除/キャンセル |
| 並び順 | v1は追加順（新しいものが上）固定、並び替え機能なし |

### v2（候補）：聞き忘れを防ぐ

**追加機能：質問チェックリスト**

| 機能 | 説明 |
|------|------|
| テンプレート提供 | 初心者向けの「聞くべきことリスト」（デフォルト10問） |
| カスタマイズ | 経験者は自分で質問を追加・削除できる |
| 入力形式 | はい / いいえ の2択タップ + 補足メモ（任意） |
| 目的 | 「比較表」のためではなく「聞き漏れ防止」 |

#### 設計方針

**見学中の物理的制約を考慮**
- 0歳児なら抱っこしながら見学することもある
- 片手でスマホを操作しながら、説明を聞いている状態
- **タップだけで完結する**のが理想

**見学中の入力**
- はい / いいえ の2択タップのみ
- 補足メモは「後から書ける」が、見学中は必須ではない
- 片手で、数秒で、1項目を埋められる

**見学後の入力**
- 補足メモの追記
- 落ち着いて両手で書ける状態で詳細を記入

**未確認の場合**
- 「未確認」ボタンは設けない
- 聞けなかったら、はい・いいえを選ばず補足だけ書くパターンもあり得る

#### デフォルト10問（入園後の毎日の負担に直結するもの）

| # | 質問 | 補足の想定 |
|---|------|-----------|
| 1 | 連絡帳はアプリですか？ | アプリ名など |
| 2 | おむつは園で処理してくれますか？ | サブスク有無、枚数など |
| 3 | お昼寝布団はコットカバーですか？ | 持ち帰り頻度など |
| 4 | 駐輪場（または駐車場）はありますか？ | 台数制限など |
| 5 | 延長保育はありますか？ | 時間、料金など |
| 6 | 手作りで用意するものはありますか？ | 何が必要かなど |
| 7 | ベビーカーは園に置いていけますか？ | 置き場所など |
| 8 | お薬の投与はしてもらえますか？ | 条件など |
| 9 | 土曜保育はありますか？ | 条件、別途申請など |
| 10 | 保護者会はありますか？ | 頻度、役割など |

#### 選定基準

**「近ければ近いほど、毎日の負担が楽であれば楽なほどいい」**

この現実的な視点から、入園後の日常生活に直結する項目を優先した。

#### 追加項目候補（カスタマイズ用）

以下は「重要だが10問に入れなかった」もの。ユーザーが必要に応じて追加できる。

| 項目 | 対象 |
|------|------|
| アレルギー対応はありますか？ | アレルギー児の保護者 |
| SIDS対策（呼吸確認、センサー）はありますか？ | 0歳児クラス |
| 給食はありますか？ | 幼稚園検討者 |
| 園庭はありますか？ | 外遊び重視の方 |
| 英語教育はありますか？ | 教育重視の方 |

※ タイムスケジュール、保育方針などは「はい・いいえ」で答えにくいため、気づきメモでの記録を推奨

### v3以降（候補）：振り返りを支援する

**追加機能：振り返りガイド**

| 機能 | 説明 |
|------|------|
| 振り返りの問いかけ | 「ここに子どもを預けたいと思えたか？」など |

| パートナー共有 | 有料機能として検討 |

## 9. 技術仕様

### データモデル（v1）

```typescript
// 園データ
interface Nursery {
  id: string;              // UUID
  name: string;            // 園名（必須）
  visitDate: string | null; // 見学日（ISO 8601形式）、未定ならnull
  memo: string;            // メモ（空文字列可）
  createdAt: string;       // 作成日時（ISO 8601形式）
  updatedAt: string;       // 更新日時（ISO 8601形式）
}

// アプリ状態
interface AppState {
  schemaVersion: number;   // データ構造バージョン（v1では1）
  nurseries: Nursery[];    // 園リスト
  hasSeenOnboarding: boolean; // オンボーディング表示済みフラグ
}
```

### localStorage仕様

| キー | 内容 |
|------|------|
| `hokatsu-techo-v1` | AppState全体をJSON文字列で保存 |

**マイグレーション方針**
- `schemaVersion` で構造変更を管理
- 読み込み時に足りないフィールドはデフォルト値で補完
- 現プロトタイプからの移行は別途検討（キー名が異なるため）

### 技術的な制約（継続）

| 項目 | 内容 |
|------|------|
| データ保存 | localStorage（ブラウザのみに保存、サーバーに送信しない） |
| オフライン対応 | PWAとして動作、圏外でも使える |
| プライバシー | 入力データは分析ツールに送信しない |

### iOS PWAの注意点

localStorageは端末や条件によって消える可能性がある（容量逼迫、長期間未使用など）。

## 10. 次のステップ

### v1（SLC）の開発

1. **情報設計** - 画面構成とデータ構造の設計
2. **Figmaデザイン** - 「片手でサッと使える」UIの設計
3. **技術実装** - Next.js + Tailwind CSSでの実装
4. **実地検証** - 実際の見学で使ってみる（Complete/Lovableの判定）

### v1の検証後

- Complete/Lovableが達成できていれば → v2の検討
- 達成できていなければ → v1の改善（機能追加ではなく体験改善）
